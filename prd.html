1. Tổng quan sản phẩm

ZesTour là hệ thống quản lý và trải nghiệm quán ăn đa nền tảng. Hệ thống gồm: Admin Web để quản trị, Backend API để đồng bộ và xử lý dữ liệu, Mobile App cho người dùng cuối, và SharedLib cho model dùng chung.

Bài toán cần giải quyết

Quy trình tiếp nhận, duyệt và đưa thông tin quán ăn lên bản đồ thường rời rạc và thủ công, gây chậm cập nhật và khó theo dõi trạng thái đăng ký của chủ quán.

Mục tiêu sản phẩm

  • Tập trung hóa dữ liệu quán ăn và POI trên một hệ thống thống nhất.
  • Cho phép seller gửi và tự quản lý đăng ký cửa hàng trên mobile.
  • Hỗ trợ admin duyệt nhanh, gán tọa độ map và tạo POI từ đăng ký.
  • Đảm bảo đồng bộ dữ liệu thời gian thực giữa admin và mobile thông qua API.
Quản trị tập trung Quy trình duyệt quán POI CRUD Phân vai người dùng

2. Persona và bên liên quan

Persona chính

  • Admin: quản lý user, quản lý quán ăn, duyệt đăng ký quán từ seller.
  • Seller: đăng ký quán, cập nhật/xóa đăng ký của chính mình, upload ảnh quán.
  • Customer: đăng ký/đăng nhập và sử dụng tính năng khám phá địa điểm (mức cơ bản).

Bên liên quan

  • Product Owner: định hướng roadmap và KPI.
  • Backend Team: API, DB, migration, data integrity.
  • Admin Web Team: dashboard và công cụ vận hành.
  • Mobile Team: onboarding, seller flow, kết nối API.

3. Phạm vi

In Scope (MVP / Build hiện tại)

  • Auth API: register, login, CRUD user, cập nhật role.
  • Store registration API: tạo, xem toàn bộ, xem theo owner, update, delete.
  • POI API CRUD và seed dữ liệu mẫu khi DB rỗng.
  • Admin Web cookie authentication và fallback policy bắt buộc đăng nhập.
  • Mobile app: login/register, gửi đăng ký cửa hàng, quản lý đăng ký của tôi.
  • Upload ảnh lên Cloudinary trước khi gửi thông tin quán từ mobile.

Out of Scope (Build hiện tại)

  • Thanh toán online, booking, loyalty, recommendation engine nâng cao.
  • Phân quyền JWT token end-to-end cho toàn hệ thống.
  • Audit trail chi tiết cho từng hành động.
  • Push notification hoặc email automation.

5. Yêu cầu chức năng (Functional Requirements)

ID Yêu cầu Mô tả Ưu tiên
FR-01 Đăng ký/Đăng nhập người dùng Hệ thống cho phép đăng ký và đăng nhập bằng username hoặc email, trả về profile và role. P0
FR-02 Quản lý user (Admin) Admin tạo/sửa/xóa user, cập nhật role (admin/seller/customer), validate username/email duy nhất. P0
FR-03 Gửi đăng ký cửa hàng Seller gửi thông tin quán (store, owner, phone, address, category, description, imageUrl). P0
FR-04 Quản lý đăng ký theo owner Seller chỉ được cập nhật/xóa đăng ký của chính ownerName đã gửi. P0
FR-05 POI CRUD Admin/backend hỗ trợ tạo, xem, sửa, xóa POI với thông tin vị trí và mô tả. P0
FR-06 Bootstrap dữ liệu Backend tự chạy migration và seed user/POI mặc định khi DB rỗng. P1
FR-07 Bảo mật phiên Admin Admin Web dùng cookie auth, fallback policy bắt buộc đăng nhập cho route mặc định. P1

6. Yêu cầu phi chức năng (NFR)

ID Nhóm Yêu cầu
NFR-01 Availability Backend và Admin chạy ổn định ở local/dev, startup không lỗi migration.
NFR-02 Performance API CRUD có thời gian đáp ứng trung bình < 800ms trong điều kiện dữ liệu dev vừa.
NFR-03 Security Validation input cho trường bắt buộc; whitelist role; owner-scope enforcement khi update/delete store.
NFR-04 Compatibility Mobile Android emulator dùng base URL 10.0.2.2 và cho phép cleartext traffic khi gọi HTTP local.
NFR-05 Maintainability Mô hình dữ liệu dùng chung qua SharedLib, route API rõ ràng, migration có version.

7. Bề mặt API hiện tại

  • PUT /api/auth/users/{id}

Store + POI

  • POST /api/store-registrations
  • GET /api/store-registrations
  • GET /api/store-registrations/owner?ownerName=...
  • PUT /api/store-registrations/{id}
  • DELETE /api/store-registrations/{id}
  • GET/POST/PUT/DELETE /api/poi

8. Chỉ số thành công (KPI)

  • Tỷ lệ gửi đăng ký cửa hàng thành công từ mobile: >= 95%.
  • Tỷ lệ admin xử lý (duyệt hoặc từ chối/xóa) đăng ký trong 24h: >= 90%.
  • Tỷ lệ lỗi API 5xx trong giai đoạn UAT: < 1%.
  • Thời gian trung bình từ lúc submit đến khi có POI hợp lệ: < 1 ngày.

9. Mốc triển khai

Mốc Phạm vi Đầu ra
M1 - Ổn định backend lõi Auth, Store Registrations, POI CRUD, migration + seed Swagger pass toàn bộ smoke test endpoint lõi
M2 - Vận hành Admin Admin login, user/store management, map-based review flow Admin xử lý được end-to-end đăng ký từ seller
M3 - Trải nghiệm Seller trên Mobile Register/login, submit/update/delete own registration, image upload Seller flow hoàn chỉnh trên Android emulator
M4 - Hardening Error handling, validation, deployment config split Bản build sẵn sàng UAT

10. Rủi ro và phương án giảm thiểu

Rủi ro Tác động Giảm thiểu
Chạy nhiều backend instance gây route mismatch Admin/mobile gọi nhầm API cũ, dữ liệu không đồng bộ Chuẩn hóa lệnh chạy, kiểm tra Swagger route trước test.
Xung đột port và file lock khi build Chậm tiến độ dev/test Stop process cũ, tách output folder tạm khi cần.
HTTP local trên Android bị chặn Mobile không kết nối backend Dùng 10.0.2.2 + cấu hình usesCleartextTraffic.
Validation model chưa đầy đủ ở mọi endpoint Dữ liệu bẩn, phát sinh lỗi nghiệp vụ Bổ sung validation rule + contract test.

11. Tiêu chí nghiệm thu (Release Gate)

  • AC-01: Seller có thể đăng ký, đăng nhập và gửi đăng ký quán thành công trên mobile.
  • AC-02: Seller chỉ sửa/xóa được bản ghi của chính mình (owner-scope pass).
  • AC-03: Admin đăng nhập và thực hiện CRUD user, xem/duyệt store registrations.
  • AC-04: POI CRUD hoạt động và dữ liệu tồn tại đúng trên DB sau restart backend.
  • AC-05: Không có lỗi blocker trong smoke test backend/admin/mobile.
Lưu ý: hiện trạng chưa có cơ chế mã hóa mật khẩu và auth token cho mobile. Đây là hạng mục ưu tiên cao cho pha hardening tiếp theo.
